AI for Research Administration

RAの現場知を、 みんなで スキルにして 共有したい。

RA Skills は、研究費申請、報告書、予算、会議運営、進捗管理、研究データ管理などの研究アドミニストレーション業務を、AI エージェントのスキルとして持ち寄るための公開リポジトリです。まず私がたたき台を置いたので、ぜひ皆さんの業務知をスキルとして共有してください。

What this project is

現場知を、共有できるスキルにするための場所です。

ここでいう RA は Research Assistant ではなく、Research Administration、つまり研究を支える実務全体を指します。各現場にある暗黙知を、入力、手順、成果物、品質基準を持つスキルとして共有できれば、ほかの現場でも再利用しやすくなるはずです。

Problem

研究者も支援者も忙しすぎる

公募検索、申請、報告、予算、会議、広報、倫理、データ管理。研究の周辺には、重要だけれど時間を食う仕事が大量にあります。

Approach

業務を「スキル」にして共有する

ただのプロンプト集ではなく、入力、手順、成果物、品質基準を持つ SKILL.md として、他の人が使える形にします。

Goal

それぞれの知見を持ち寄る

GitHubでPRできる人はスキルを追加し、難しければフィードバックだけでも送れるようにします。

Current draft skills

今ある16本は、共有のためのサンプルでもあります。

現在のリポジトリには16本のスキルがあります。研究費、公募、申請書、内部締切、会議、進捗、報告書、予算、DMP、倫理、イベント運営などを題材に、業務知をどうスキル化できるかを試しています。

01

探す

`/funding-search` と `/policy-monitor` で、公募・政策・機会を見つける。

02

申請する

`/grant-proposal-review` と `/budget-planning` で、申請書と予算案を整える。

03

締切を逆算する

`/internal-deadline-tracker` で、外部締切、学内締切、部局承認、e-Rad承認を整理する。

04

運営する

`/project-meeting-ops` と `/project-progress-tracker` で、会議、ToDo、月次進捗、リスクを追う。

05

守る・広げる

`/ethics-review-support`、`/research-data-management`、`/event-organizer` で周辺実務を支援する。

Reality check

たぶん、RAの仕事はもっと広い。

MEXT / CRAMS の URA スキル標準 22 業務に照らすと、今の repo はまだ入口です。今回、会議運営、予定調整、進捗管理、内部・対外折衝の入口は追加しましたが、契約、知財、安全管理、輸出管理、研究広報など、実務に詳しい方に確認したい領域が多く残っています。

現時点の見立て

書類作成支援だけでは、RA業務の全体像には届かなそうです。だからこそ、各現場の「この仕事はこう進める」という知見を持ち寄って、スキルとして増やしたいです。

Prototype test

スキル化すると、業務知を試せる形になります。

JST公募準備と、採択後3か月の月次進捗会議を想定して試行しました。こうして試せる形にしておくと、「その手順は違う」「この確認先が抜けている」といった現場からの修正を受けやすくなります。

Pre-award

公募準備の逆算

funding-search、grant-proposal-review、budget-planning、internal-deadline-tracker で、JSTさきがけ応募支援のモデルケースを作りました。

Post-award

月次進捗会議

project-progress-tracker と project-meeting-ops で、遅延、倫理確認、RA採用、会議後ToDoを担当者・期限付きで出しました。

Result

まだ改善余地あり

会議・締切は入口として書けましたが、予算執行、成果台帳、契約・知財・輸出管理は現場知のコントリビューションが必要です。

Contribute

あなたの現場の、1つの手順を教えてください。

GitHubで編集できる方は Pull Request でスキルを追加してください。実際に PR #1 では、rmaruy3 さんが event-organizer スキルを追加してくれました。難しければ、右下のフィードバックから、業務名、普段の手順、確認先、よくある失敗を数行送ってもらえるだけでも十分です。

What

業務名と成果物

例: 学内締切確認、共同研究契約の入口、イベント後アンケート。最後に何が出ればよいかを教えてください。

How

手順と確認先

最初に見る書類、誰に聞くか、学内と部局で違う点、締切前に必ず確認することを書いてください。

Risk

失敗例と注意点

この判断はAIに任せない、ここは事務・倫理・知財に確認する、という境界線が一番助かります。

Draft roadmap

次に試したいこと。

5月20日の勉強会では、完成品の発表というより、現場の人と一緒に「RAの業務知をスキルとして共有する場」を作るためのたたき台として共有します。使えそうなところ、ズレているところ、足したいスキルを教えてください。

Tier 0

会議・進捗・締切

`project-meeting-ops`、`project-progress-tracker`、`internal-deadline-tracker`、`stakeholder-coordination` は、投稿例としても見られるようにしています。

Tier 1

専門部署へつなぐ

契約、知財、輸出管理、研究広報、安全管理は、判断代行ではなく論点整理に限るべきかもしれません。

Tier 2

研究戦略へ広げる

政策情報、研究力分析、公募情報をどう扱うべきかも、実務のニーズを聞きながら考えたいです。
