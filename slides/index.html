Act II · Definition
03 / 13
What is a skill?

現場の暗黙知を、
共有できる
業務手順にする。

入力、手順、成果物、品質基準を SKILL.md として書く。個人や部署に閉じたノウハウを、ほかの人も試せる形にする。

Input → Procedure → Output → Quality Gate
Skill = 現場知を共有するための小さな単位

スキル化の例

こんな業務知を、スキルとして共有できます。

funding-search
出せそうな公募を探す
公募一覧、適合度、応募資格、公式URL、締切、確認事項。
internal-deadline-tracker
締切を逆算する
外部締切、学内締切、部局承認、e-Rad承認、担当者別ToDo。
grant-proposal-review
申請書をレビューする
審査観点ごとの弱点、改善案、確認すべき根拠。
project-meeting-ops
会議を回す
日程調整文、アジェンダ、議事録、会議後ToDo。
project-progress-tracker
進捗とリスクを追う
月次サマリー、マイルストーン、リスク表、成果登録表。
stakeholder-coordination
関係者を整理する
関係者マップ、合意事項、未決事項、確認依頼メール。
皆さんの業務も、この粒度で共有できるかもしれない

共有したい知見の形

「こう進める」を、
確認表やToDoに落とす。

公募準備
「この公募、間に合う？」に対して、学内締切・推薦枠・共同機関承諾・e-Rad承認を未確認事項として出す。
Pre-award
月次会議
「来週の定例会を準備して」に対して、論点、アジェンダ、議事録テンプレ、会議後ToDoを作る。
Ops
専門部署
「これは倫理・契約・知財が絡む？」に対して、断定せず、確認先と確認依頼文に変換する。
Govern
判断そのものより、確認先・未決事項・次アクションを共有したい

09 / 13
実際に6つのスキルをつないでみる

若手PIのJST応募支援を、
1本の作業列にする。

01
探す
funding-searchで、JSTさきがけを第一候補にする。
02
位置づける
policy-monitorで、AIと研究開発プロセス革新の文脈を見る。
03
磨く
申請書レビューで、独自性、評価指標、現場検証を補う。
04
間に合わせる
予算、学内締切、e-Rad承認を高リスク案件として逆算する。
成果物は examples/model-case-pre-award-jst-presto.md にまとめました
Model Case
実際にスキルを順番につないで、JSTさきがけ応募支援のモデルケースを作りました。公募候補、政策文脈、申請書レビュー、仮予算、内部締切、e-Radチェックリストまで出しています。大事なのは、AIが判断を代替するのではなく、確認事項と次アクションを見える形にすることです。

お願い

あなたの知見を、
スキルとして
共有してください。

01
PRできる人はPRで
`skills/` に新しい SKILL.md を追加してください。実際に event-organizer はPRで追加されました。
02
難しければフィードバックで
業務名、普段の手順、確認先、危ない判断だけでも送ってください。
03
現場知を共有財に
各大学・部署にあるノウハウを、AI時代に再利用できる形にしたい。
t46.github.io/ra-skills github.com/t46/ra-skills
GitHub / Issue / メール、どの形でも歓迎です
End
最後は、ぜひ皆さんの知見をスキルとして共有してください、で終わりたいと思っています。実際にPR #1では、私以外の方がevent-organizerスキルを追加してくれました。GitHubでPRできる方は直接追加してほしいですし、難しければ右下のフィードバックから、業務名や普段の手順、確認先、危ない判断を送ってもらえるだけでも十分です。
