# 研究プロジェクト進捗管理

## What This Does

採択後の研究プロジェクトについて、研究計画、マイルストーン、予算、成果、リスク、報告期限を月次で整理する。進捗報告書の直前に慌てて材料を集めるのではなく、日常の定例会議と成果管理から報告・評価に使えるログを作る。

- **対象ユーザー**: URA、研究支援担当者、PI、プロジェクトマネージャー、部局事務
- **解決する課題**: 研究進捗、成果、遅延理由、対応方針が散在し、報告・評価時に再構成が必要になる
- **期待される効果**: 遅延の早期発見、報告書作成の負担軽減、評価対応の品質向上

## Required Inputs

- **研究課題情報**: 課題名、研究期間、資金配分機関、研究代表者、共同研究者
- **当初計画**: 年度計画、研究項目、マイルストーン、成果目標
- **現在の状況**: 完了作業、進行中作業、遅延、課題、予算執行状況
- **成果情報**: 論文、学会発表、特許、データ公開、プレスリリース、受賞など
- **次の締切**: 報告書、評価会、予算執行、倫理更新、契約更新など
- **前回ログ**（任意）: 前月の進捗サマリー、ToDo、未決事項

## What It Produces

1. **月次進捗サマリー**: PIや関係者に共有できる短い要約
2. **マイルストーン表**: 計画、実績、遅延、対応方針を一覧化
3. **課題・リスク一覧**: 影響、期限、担当、エスカレーション先を整理
4. **成果登録表**: 報告書に転記できる成果一覧
5. **PI確認事項**: 研究者に確認すべき事項と回答期限

## 手順

### 1. 当初計画を作業単位に分解する

申請書や採択時計画から、年度・四半期・月次で確認できる粒度へ分解する。

| 研究項目 | 計画上の到達点 | 期限 | 主担当 | 成果物 |
|----------|----------------|------|--------|--------|
| 例: データ収集 | 100件の調査完了 | 2026-09 | A班 | 調査データ |

「研究を進める」のような抽象タスクは、確認可能な成果物に変換する。

### 2. 現在の進捗を4分類する

各研究項目を次の4つに分ける。

| 状態 | 判断基準 | 次アクション |
|------|----------|--------------|
| 完了 | 成果物があり、確認済み | 成果登録表に入れる |
| 順調 | 期限内に完了見込み | 次回確認日を置く |
| 要注意 | 遅延・不足があるが回復可能 | 対策と期限を決める |
| 要エスカレーション | 研究計画、予算、契約、倫理に影響 | PI・専門部署・FA相談を検討 |

### 3. リスクを整理する

リスクは「困っていること」ではなく、影響と対応期限が分かる形で書く。

```markdown
## リスク・課題

| No | 内容 | 影響 | 対応方針 | 担当 | 期限 | エスカレーション先 |
|----|------|------|----------|------|------|--------------------|
```

典型的なリスク:

- 人員採用の遅れ
- 倫理審査・契約・輸出管理・安全管理の未了
- 共同機関からのデータ・成果物の遅れ
- 予算執行の過不足、年度末納品リスク
- 論文投稿、知財、広報の順序問題

### 4. 成果を登録する

報告書で再利用できるよう、成果は発生時点で記録する。

| 区分 | 必須項目 | 注意 |
|------|----------|------|
| 論文 | 著者、タイトル、誌名、年、DOI、査読、OA | 報告対象期間を確認 |
| 学会 | 発表者、題目、会議名、日付、発表種別 | 招待/一般を区別 |
| 特許 | 発明名称、出願番号、出願日、公開可否 | 論文・広報前に知財確認 |
| データ | データ名、リポジトリ、公開日、DOI/URL | DMPと整合 |
| 広報 | タイトル、媒体、公開日、URL | 機関広報確認 |

### 5. 月次サマリーを作成する

PIや事務が読める短さで、結論から書く。

```markdown
# 月次進捗サマリー

- 対象期間:
- 全体状況: 順調 / 要注意 / 要エスカレーション
- 今月の主な進捗:
- 主な成果:
- 遅延・リスク:
- PIに確認したいこと:
- 次月までのToDo:
```

### 6. 報告・評価へ接続する

報告書作成時は、このスキルの成果物を `progress-report` に渡す。

- 月次進捗サマリー → 研究実績の概要
- マイルストーン表 → 計画との差異分析
- 成果登録表 → 成果一覧
- リスク一覧 → 遅延理由と今後の推進方策

## 判断基準

| 判断項目 | 見るポイント | 対応 |
|----------|--------------|------|
| 進捗 | 当初計画のどの項目に対応するか | 計画項目IDを付ける |
| 遅延 | 期限、成果物、外部依存に影響するか | 要注意/要エスカレーションに分ける |
| 成果 | 報告書・広報・知財・DMPに使えるか | 登録表に証跡を残す |
| 予算 | 研究進捗と執行が乖離していないか | `budget-execution`へ引き渡す |
| 専門判断 | 契約、倫理、知財、安全、輸出管理に関係するか | 専門部署確認をToDo化する |

## よくあるミスと対策

| ミス | 対策 |
|------|------|
| 進捗を「順調です」で終える | 研究項目、成果物、次回確認日を入れる |
| 報告時期に成果一覧を初めて集める | 月次で成果登録表を更新する |
| 遅延理由が曖昧 | 原因、影響、回復策、期限を分ける |
| 予算執行と研究進捗を別管理する | 月次で予算リスクを確認する |
| 専門部署相談が遅れる | リスク表にエスカレーション先を入れる |

## 品質基準

- 当初計画と現在の進捗が対応づけられていること
- 遅延やリスクに、影響、担当、期限、確認先があること
- 成果一覧が報告書に転記できる粒度になっていること
- PIに確認すべき事項が明確で、回答期限があること
- `progress-report`, `budget-execution`, `research-data-management` へ引き渡せること

## Available Tools

- **Read**: 申請書、採択通知、進捗メモ、予算表、成果リストの読み込み
- **Write**: 月次サマリー、マイルストーン表、リスク表、成果登録表の作成
- **WebSearch** / **WebFetch**: FAの報告期限、評価要件、成果公開要件の確認
- **Bash**: CSV/Markdown表の整形、期限の並べ替え、文字数確認
