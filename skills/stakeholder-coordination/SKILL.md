# 研究プロジェクト関係者調整

## What This Does

研究費申請・採択後運営・共同研究・評価対応に関わる学内外の関係者を整理し、誰に何を確認し、どの合意が済み、何が未決かを見える化する。内部折衝・対外折衝を「メールを送る」だけでなく、責任分担、確認先、期限、エスカレーション先を含む調整計画にする。

- **対象ユーザー**: URA、研究支援担当者、PI、プロジェクトマネージャー、部局事務
- **解決する課題**: 関係者が多く、合意事項・未決事項・確認先が散在する
- **期待される効果**: 確認漏れ、責任分担の曖昧さ、専門部署への相談遅れを減らす

## Required Inputs

- **調整の目的**: 申請準備、共同研究立上げ、採択後キックオフ、契約変更、評価対応など
- **関係者情報**: 氏名、所属、役割、連絡先、決定権限
- **現在の状況**: 決まっていること、未決事項、懸念、期限
- **関係する制度・部署**: 経理、契約、知財、倫理、安全、輸出管理、情報セキュリティなど
- **期限**: 公募締切、契約開始日、評価会、報告期限、学内締切
- **既存資料**（任意）: 申請書、研究計画、予算表、契約案、議事録、メール

## What It Produces

1. **関係者マップ**: 役割、関心、決定権限、連絡優先度を整理
2. **合意事項・未決事項リスト**: 決まったことと確認が必要なことを分離
3. **確認依頼文面**: PI、共同機関、専門部署へ送る依頼メール
4. **調整ToDo表**: 担当、期限、確認先、状態
5. **エスカレーション案**: 遅延・対立・制度判断が必要な場合の相談先

## 手順

### 1. 調整のゴールを定義する

調整の目的を、成果物または判断として書く。

```markdown
調整ゴール:
- いつまでに:
- 誰の合意が必要で:
- 何を決める/提出する:
```

例:

- 申請書提出前に共同研究者の分担、予算、承諾を確定する
- 採択後キックオフまでに契約、経理、倫理、DMPの確認先を決める
- 評価会前に成果、遅延理由、今後の対応方針をPIと合意する

### 2. 関係者を分類する

| 分類 | 典型例 | 確認すること |
|------|--------|--------------|
| 研究側 | PI、分担者、共同研究者 | 研究計画、役割、成果物、期限 |
| 学内事務 | 部局事務、研究推進課、経理 | 学内締切、承認、予算、証憑 |
| 専門部署 | 契約、知財、倫理、安全、輸出管理、情報基盤 | 制度・規程・リスク判断 |
| 学外関係者 | 共同機関、企業、FA、委託先 | 契約、分担、提出物、連絡窓口 |
| 意思決定者 | 部局長、センター長、研究代表者 | 方針、優先順位、例外対応 |

### 3. 関係者マップを作る

```markdown
| 関係者 | 所属 | 役割 | 決定権限 | 関心/懸念 | 依頼事項 | 期限 |
|--------|------|------|----------|-----------|----------|------|
```

決定権限が不明な場合は、早めに確認する。窓口担当と決裁者は別であることが多い。

### 4. 合意事項と未決事項を分ける

```markdown
## 合意事項
| No | 内容 | 合意者 | 日付 | 根拠資料 |
|----|------|--------|------|----------|

## 未決事項
| No | 内容 | 何が不足しているか | 確認先 | 担当 | 期限 | 影響 |
|----|------|--------------------|--------|------|------|------|
```

未決事項は「誰かに確認」ではなく、確認先と期限を入れる。

### 5. 専門部署へ渡す論点を整理する

RAが最終判断してはいけない領域は、確認依頼に変換する。

| 領域 | 典型論点 | 確認先 |
|------|----------|--------|
| 契約 | NDA、共同研究、受託、データ提供、再委託 | 契約担当、法務 |
| 知財 | 発明、共同出願、論文投稿前公開、ライセンス | 知財担当 |
| 倫理 | 人対象研究、同意、個人情報、二次利用 | 倫理審査窓口 |
| 安全 | 薬品、遺伝子組換え、病原体、放射線 | 安全管理担当 |
| 輸出管理 | 技術提供、海外送付、外国機関、用途 | 輸出管理担当 |
| 情報管理 | 機微情報、クラウド利用、アクセス権限 | 情報セキュリティ担当 |

### 6. 確認依頼文を作る

```markdown
件名: [確認依頼] {プロジェクト名} {確認事項}

{プロジェクト名} について、下記の点をご確認いただけますでしょうか。

背景:
- {背景}

確認したい事項:
- {事項1}
- {事項2}

関連資料:
- {資料}

希望回答期限:
- {日付}

こちらで把握している未決事項:
- {未決事項}
```

### 7. 調整状況を更新する

調整ToDoは状態を持たせる。

| 状態 | 意味 |
|------|------|
| 未着手 | まだ依頼していない |
| 依頼済 | 相手に依頼済み |
| 回答待ち | 回答待ち |
| 要追加確認 | 回答だけでは足りない |
| 完了 | 合意または確認完了 |
| エスカレーション | 期限・判断・対立で上位相談が必要 |

## 判断基準

| 判断項目 | 見るポイント | 対応 |
|----------|--------------|------|
| 決定権限 | 誰が最終判断できるか | 窓口と決裁者を分ける |
| 未決事項 | 何が足りず止まっているか | 確認先と期限を置く |
| 専門判断 | 法務・知財・倫理・安全・輸出管理に関係するか | 専門部署へ渡す |
| 期限 | 外部締切・内部締切に影響するか | 優先順位を上げる |
| 証跡 | 合意がメール・議事録・資料に残っているか | 根拠資料を記録する |

## よくあるミスと対策

| ミス | 対策 |
|------|------|
| 関係者リストだけ作って調整が進まない | 依頼事項、期限、状態を入れる |
| 窓口担当を決裁者だと思い込む | 決定権限を別項目で確認する |
| 専門部署へ抽象的に相談する | 背景、確認事項、資料、希望期限を揃える |
| 合意事項が口頭だけで残る | 議事録またはメールで証跡化する |
| 未決事項が積み残る | 状態を更新し、期限超過はエスカレーションする |

## 品質基準

- 関係者の役割、決定権限、依頼事項、期限が明示されていること
- 合意事項と未決事項が分離され、根拠資料が残っていること
- 専門部署へ確認すべき論点をRAが断定していないこと
- 調整ToDoに状態があり、期限超過時のエスカレーション先があること
- 研究者、事務、共同機関のどこで止まっているかが一目で分かること

## Available Tools

- **Read**: 申請書、契約案、議事録、メール、予算表の読み込み
- **Write**: 関係者マップ、未決事項表、確認依頼文、調整ToDoの作成
- **WebSearch** / **WebFetch**: FA・機関公開情報、制度窓口、最新要件の確認
- **Bash**: 関係者表、ToDo表、期限表の整形
