# 研究成果台帳管理

## What This Does

研究プロジェクトの成果を、報告書作成時だけでなく日常的に記録・更新する。論文、学会発表、特許、データ公開、ソフトウェア、プレスリリース、受賞、人材育成、アウトリーチを台帳化し、進捗報告、最終報告、評価対応、広報、次期申請に使える形にする。

- **対象ユーザー**: URA、研究支援担当者、PI、プロジェクトマネージャー、広報・知財担当
- **解決する課題**: 成果情報が研究者の記憶、メール、researchmap、報告書案に散在する
- **期待される効果**: 報告漏れ、謝辞漏れ、知財・広報確認漏れを減らし、成果を次の資金獲得へつなげる

## Required Inputs

- **プロジェクト情報**: 課題名、制度名、研究期間、研究代表者、成果報告要件
- **成果候補**: 論文、発表、特許、データ、ソフトウェア、広報、受賞、教育実績
- **根拠資料**: DOI、URL、プログラム、投稿状況、受理通知、広報記事、特許相談メモ
- **確認先**: PI、分担者、知財、広報、DMP担当、経理、FA
- **報告期限**（任意）: 年次報告、評価会、最終報告、学内報告

## What It Produces

1. **研究成果台帳**: 成果種別、状態、根拠、関連課題、確認先、報告可否を整理
2. **報告書用成果リスト**: FA報告、学内報告、評価会向けに整えた一覧
3. **広報・知財確認リスト**: 公開前確認が必要な成果を抽出
4. **未回収情報リスト**: DOI、謝辞、共同著者確認、データ公開URLなど不足項目
5. **次期申請向け成果要約**: 研究実績として使える短い説明

## 手順

### 1. 成果種別を定義する

制度・機関の報告様式に合わせて分類する。分類が未確認なら、一般分類として扱う。

| 種別 | 例 | 注意点 |
|------|----|--------|
| 論文 | 査読付き論文、プレプリント、総説 | DOI、謝辞、課題番号、OA状態 |
| 学会 | 口頭、ポスター、招待講演 | 開催日、発表者、国際/国内 |
| 知財 | 発明届、特許出願、ライセンス | 公開前に知財担当へ確認 |
| データ | データセット、DMP、リポジトリ | 公開範囲、個人情報、ライセンス |
| ソフトウェア | GitHub、パッケージ、ツール | ライセンス、依存関係、保守 |
| 広報 | プレス、Web記事、取材 | 広報確認、公開日、表現 |
| 人材育成 | 学生、RA、博士、講習 | 報告対象か確認 |
| アウトリーチ | 講演、展示、勉強会 | 対象者、参加者数、URL |

### 2. 台帳を作る

```markdown
| ID | 種別 | 成果名 | 状態 | 日付 | 関係者 | 根拠URL/資料 | 課題番号/謝辞 | 確認先 | 報告可否 | 未決事項 |
|----|------|--------|------|------|--------|---------------|---------------|--------|----------|----------|
```

状態は `予定 / 投稿中 / 採択 / 公開済 / 確認中 / 報告済 / 報告対象外` で管理する。

### 3. 報告・広報・知財の分岐をつける

成果ごとに、どこへ引き渡すかを明示する。

- 報告書に使う: `progress-report`
- 広報に使う: `research-pr-support`
- 知財確認が必要: `ip-disclosure-triage`
- データ公開が必要: `research-data-management`
- 次期申請に使う: `grant-proposal-review`

### 4. 未回収情報をPIへ確認する

```markdown
件名: [確認依頼] {プロジェクト名} 成果台帳の未確認項目

成果台帳更新のため、下記をご確認ください。

確認したい成果:
- {成果名}

不足情報:
- DOI / URL:
- 謝辞・課題番号:
- 公開可否:
- 知財・広報確認の要否:

希望回答期限:
- {日付}
```

### 5. 報告書用に整える

報告書に転記する前に、制度が求める形式、文字数、成果種別、課題番号、謝辞表記を確認する。

## 判断基準

| 判断項目 | 見るポイント | 対応 |
|----------|--------------|------|
| 根拠 | DOI、URL、通知、資料があるか | 根拠欄に記録する |
| 報告可否 | FA・機関の報告対象か | 要確認なら未決にする |
| 公開リスク | 知財、個人情報、契約に触れるか | 専門部署へ確認 |
| 謝辞 | 課題番号・助成表記が正しいか | 公式表記を確認 |
| 再利用 | 次期申請や広報で使えるか | 短い成果要約を作る |

## よくあるミスと対策

| ミス | 対策 |
|------|------|
| 報告直前に成果を集める | 月次で台帳更新日を置く |
| 論文だけを成果として扱う | データ、ソフト、広報、人材育成も分類する |
| 知財確認前に公開情報を書きすぎる | 発明可能性があれば知財担当へ確認 |
| DOIや謝辞が抜ける | 未回収情報リストを作る |

## 品質基準

- 成果種別、状態、根拠、確認先、報告可否が分かること
- 報告書、広報、知財、DMP、次期申請への引き渡し先があること
- 公開前確認が必要な成果をRAが勝手に公開判断していないこと
- 月次更新に使える表になっていること

## Available Tools

- **Read**: 報告書、論文リスト、researchmap、KAKEN、OpenAlex、議事録の読み込み
- **Write**: 成果台帳、未回収情報リスト、報告書用成果一覧の作成
- **WebSearch** / **WebFetch**: DOI、公開URL、公式成果ページの確認
- **Bash**: CSV、Markdown表、重複チェックの整形
