# 採択後キックオフ支援

## What This Does

研究費や共同研究が採択・開始決定した直後に、研究者、部局事務、経理、契約、知財、倫理、安全、情報基盤、共同機関との初期確認を整理する。採択通知を受け取ってから研究を始めるまでに必要な論点を洗い出し、誰がいつ何を確認するかを明確にする。

- **対象ユーザー**: URA、研究支援担当者、PI、部局事務、プロジェクトマネージャー
- **解決する課題**: 採択後に必要な確認が散在し、契約・経理・倫理・DMP・共同機関対応が後手になる
- **期待される効果**: 研究開始遅延、支出不可、契約漏れ、専門部署相談遅れを減らす

## Required Inputs

- **採択情報**: 制度名、FA、課題名、研究期間、採択額、採択通知日
- **体制**: PI、分担者、共同機関、事務担当、専門部署候補
- **研究計画・予算**: 申請書、採択時コメント、予算内訳、年度配分
- **開始希望日・制約**: 契約開始日、研究開始日、初回支出予定、報告期限
- **関連リスク**（任意）: 人対象研究、個人情報、海外連携、企業連携、発明可能性、安全管理

## What It Produces

1. **採択後キックオフ確認表**: 経理、契約、倫理、DMP、知財、安全、広報、共同機関の確認事項
2. **初回会議アジェンダ**: 決めること、確認すること、専門部署へ渡すこと
3. **担当者別ToDo表**: 担当、期限、確認先、状態、根拠資料
4. **専門部署への確認依頼文**: 背景、確認事項、希望回答期限を含む文面
5. **初期リスクメモ**: 研究開始前に止めるべき事項、要確認事項、次アクション

## 手順

### 1. 採択条件を公式資料で確認する

採択通知、交付内定通知、委託研究契約書案、公募要領、事務処理説明書を確認する。未確認の制度情報は断定しない。

```markdown
確認した根拠:
- 採択通知:
- 事務処理説明書:
- 公募要領:
- 所属機関通知:
```

### 2. 研究開始前に必要な領域を分解する

| 領域 | 主な確認事項 | 確認先 |
|------|--------------|--------|
| 経理 | 予算配分、費目、支出開始日、間接経費、発注期限 | 経理、研究推進課 |
| 契約 | 共同研究、委託、再委託、データ提供、NDA | 契約担当、法務 |
| 倫理 | 人対象研究、同意、個人情報、二次利用 | 倫理審査窓口 |
| DMP | データ管理計画、公開、保存、アクセス権限 | RDM担当、図書館、情報基盤 |
| 知財 | 発明可能性、論文投稿前確認、共同出願 | 知財担当 |
| 安全 | 薬品、遺伝子組換え、病原体、放射線、動物 | 安全管理担当 |
| 共同機関 | 分担、契約、請求、報告、成果物 | 共同機関窓口 |
| 広報 | 採択発表、Web掲載、プレス対応 | 広報担当 |

### 3. キックオフ会議を設計する

会議は「説明」ではなく、開始前に止まりそうな点を特定する場にする。

```markdown
# 採択後キックオフ アジェンダ

1. 採択条件と研究期間の確認
2. 初年度予算と支出開始条件
3. 共同機関・契約・再委託の確認
4. 倫理・安全・DMP・情報管理の要否確認
5. 成果・報告・広報・知財の管理方法
6. 初月ToDoと次回会議
```

### 4. 担当者別ToDoに落とす

```markdown
| No | タスク | 担当 | 確認先 | 期限 | 根拠資料 | 状態 |
|----|--------|------|--------|------|----------|------|
```

状態は `未着手 / 依頼済 / 回答待ち / 要追加確認 / 完了 / 保留` で管理する。

### 5. 専門部署への確認依頼を作る

RAが判断できない事項は、背景と資料を添えて確認依頼にする。

```markdown
件名: [確認依頼] {課題名} 採択後開始前確認

{制度名} に採択された {課題名} について、研究開始前に下記をご確認ください。

背景:
- 研究期間:
- 関係機関:
- 関連資料:

確認したい事項:
- {確認事項1}
- {確認事項2}

希望回答期限:
- {日付}
```

## 判断基準

| 判断項目 | 見るポイント | 対応 |
|----------|--------------|------|
| 支出開始 | いつから何を支出できるか | 経理・制度資料で確認 |
| 契約 | 研究開始前に契約が必要か | 契約担当へ確認 |
| 倫理・安全 | 承認前に開始できない作業があるか | 専門窓口へ確認 |
| 共同機関 | 分担・請求・報告が決まっているか | 窓口と期限を設定 |
| 成果管理 | 論文、知財、データ、広報の記録方法があるか | 成果台帳を作る |

## よくあるミスと対策

| ミス | 対策 |
|------|------|
| 採択通知だけ見て開始できると思う | 交付条件・契約・支出開始条件を分けて確認する |
| 共同機関との契約確認が遅れる | キックオフ時点で契約・請求・成果物をToDo化する |
| 倫理・安全・DMPを後回しにする | 研究開始前に止まる事項として最初に確認する |
| 初回会議の後に誰も動かない | 担当者、期限、確認先、状態を残す |

## 品質基準

- 採択条件、機関ルール、専門部署確認が分かれていること
- 研究開始前に止めるべき事項を見落としていないこと
- すべての未決事項に担当者、確認先、期限があること
- `budget-execution`, `project-meeting-ops`, `research-output-registry` へ渡せること

## Available Tools

- **Read**: 採択通知、公募要領、事務処理説明書、申請書、予算表の読み込み
- **Write**: キックオフ確認表、アジェンダ、ToDo表、確認依頼文の作成
- **WebSearch** / **WebFetch**: FA公式資料、所属機関公開情報の確認
- **Bash**: 表、期限表、担当者一覧の整形
