# RA Skills 実務適合性監査

監査日: 2026-05-18

## 監査基準

この監査は、既存の `skills/*/SKILL.md` が日本の大学 URA / 研究支援実務で使えるかを判定するためのもの。

主な基準:

- **実務適合性**: MEXT / CRAMS の URA スキル標準 22 業務、国内大学の URA 業務体系と対応しているか。
- **正確性**: 制度、金額、締切、システム操作、審査基準を断定しすぎていないか。
- **実行可能性**: 入力、手順、出力が実務の順番に沿い、Claude Code が成果物を作れる粒度か。
- **境界管理**: 所属機関差、年度差、専門部署への確認が必要な事項を明示しているか。
- **陳腐化耐性**: 公募、e-Rad、倫理指針、DMP、輸出管理など変わりやすい情報を固定値として抱え込みすぎていないか。

判定:

- **A**: 現場投入可。軽微な出典追加で使える。
- **B**: 軽微修正で可。構造はよいが確認先・出典・境界管理が不足。
- **C**: 大幅改修必要。実務の順番、制度理解、スコープに問題がある。
- **D**: 分割または撤回検討。

## 監査サマリー

| Skill | 判定 | 実務対応 | 主な問題 | 次アクション |
|---|---:|---|---|---|
| `policy-monitor` | B | 研究戦略推進支援: 政策情報等の調査分析 | 情報源はよいが、大学内への影響展開と保存形式が弱い | 政策 digest の出力テンプレート、学内共有先、重要度判定根拠を追加 |
| `funding-search` | B | プレアワード: 外部資金情報収集 | 公募情報源が不足し、締切・内部締切・重複制限の検証手順が薄い | FA 別チェック、公式 URL 優先、内部締切逆算を強化 |
| `grant-proposal-review` | C | プレアワード: 申請資料作成支援 | 審査基準や配点を一般化しすぎ。公募別の公式審査観点確認が必須 | KAKENHI / JST / AMED など公募別 reviewer mode に分割または条件分岐 |
| `budget-planning` | C | 申請資料作成支援の予算計画 | 科研費の費目説明に誤解リスクがあり、機関会計・FA ルール差が薄い | 固定ルールを削り、FA 要領・所属機関規程確認型に改修 |
| `budget-execution` | B | ポストアワード: プロジェクト予算管理 | 実務フローは有用。科研費・補助金・委託費など資金種別差をさらに明確化すべき | 発注前確認メモと資金種別別の確認表を追加 |
| `progress-report` | B | ポストアワード: 報告書作成業務 / 評価対応 | 報告書ドラフト支援として有用だが、評価委員会・定例進捗管理との接続が弱い | 進捗管理ログ、成果管理台帳、評価対応資料との連携を追加 |
| `research-metrics` | B | 研究戦略推進支援: 研究力の調査分析 | 指標の限界は書けているが、機関 IR / 部局戦略向けの出力が弱い | 個人支援版と組織 IR 版を分ける |
| `era-guide` | C | プレ/ポスト横断のシステム操作支援 | UI 操作、利用時間、ファイル上限など陳腐化しやすい固定値が多い | 公式マニュアル確認を必須にし、操作名の断定を減らす |
| `ethics-review-support` | B | 関連専門: 倫理・コンプライアンス | 生命・医学系と社会科学系の制度差、個人情報保護、機関委員会差の整理が必要 | 要否判定ではなく「論点整理 + 確認依頼文」中心へ寄せる |
| `research-data-management` | A | 研究データ管理 / ポストアワード実務 | DMP と運用タスクの接続がよい。公的資金の DMP 要件への出典追加が必要 | 内閣府研究 DX / FAIR / 機関リポジトリ確認を追加 |
| `event-organizer` | B | 関連専門: イベント開催関連 / 研究広報 | 実務成果物は具体的。ただし Luma + Zoom 無料開催に固定しすぎ | 汎用イベント運営版とオンライン勉強会版に分ける |
| `create-ra-skill` | B | メタスキル | 初期スキル作成には有用だが、監査基準・公式ソース確認がない | 生成後に `docs/skill-audit.md` 基準でセルフチェックさせる |

## 横断的な問題

### 1. URA 業務の中心である調整・運営が薄い

MEXT スキル標準では、プレアワードに「内部折衝」「対外折衝」、ポストアワードに「運営ミーティング」「進捗管理」「予算管理」が明示されている。現行スキルは文書作成・情報検索に寄っており、予定調整、会議体設計、議事録、タスク追跡、関係部署との調整が薄い。

追加候補:

- `project-meeting-ops`
- `stakeholder-coordination`
- `internal-deadline-tracker`
- `project-progress-tracker`

### 2. 制度情報を固定しすぎるリスクがある

`era-guide`, `budget-planning`, `grant-proposal-review` は特に、UI、締切、配点、費目、ファイルサイズ、利用時間のような変更される情報を本文に持っている。スキルは最新公式情報を確認する手順に寄せ、固定値は例示として扱う必要がある。

### 3. 所属機関差の扱いが不足している

研究費の執行、倫理審査、契約、知財、旅費、謝金、発注権限、検収は大学ごとの規程差が大きい。現行スキルは「所属機関に確認」と書くものの、何を誰に確認するかが不足している。

改修方針:

- 必ず「確認先」「確認する論点」「確認期限」を出力に含める。
- 判断結果は `可 / 要確認 / 不可` に分け、`要確認` を失敗扱いしない。
- 制度・規程・契約・倫理・会計・情報セキュリティを混ぜない。

### 4. 研究プロジェクトの lifecycle がつながっていない

現行スキルは個別 task としては成立しているが、公募発見、申請準備、採択後キックオフ、進捗管理、予算管理、成果管理、報告、広報、知財、終了処理がつながっていない。

追加候補:

- `award-kickoff`
- `project-closeout`
- `research-output-registry`
- `evaluation-board-ops`

## スキル別の詳細指摘

### `policy-monitor`

良い点:

- 政府、FA、審議会を情報源にしており、URA の政策情報等の調査分析に合う。
- 重要度と推奨アクションの形式がある。

不足:

- 学内にどう流すか、誰に知らせるか、次回確認日をどう置くかが弱い。
- 政策情報と公募情報が混ざりやすい。

改修:

- 出力に「学内影響先」「要対応部署」「次回ウォッチ日」を追加。
- `funding-search` への引き渡し条件を追加。

### `funding-search`

良い点:

- 研究分野、キャリアステージ、金額、期限の入力が実務に合う。
- 公募一覧、詳細、スケジュール案という成果物は使いやすい。

不足:

- 機関内限定公募、部局内推薦、公募件数制限の扱いが弱い。
- FA 公式ページと二次情報の優先順位が明示されていない。

改修:

- 公式ソース優先順位を `FA公式 > e-Rad > 所属機関通知 > 二次情報` とする。
- 公募ごとに「学内推薦の有無」「内部締切」「重複制限」「応募資格の証跡」を追加。

### `grant-proposal-review`

良い点:

- ストーリーライン、審査基準、具体的改善案の構造は実用的。

不足:

- 科研費、JST、AMED、民間助成で審査観点が大きく異なるため、共通配点のように見せるのは危険。
- 分野別審査慣行、若手/大型/拠点型の違いが不足。

改修:

- 最初に公募要領・審査要領・様式を読み込むことを必須化。
- `KAKENHI mode`, `JST mode`, `AMED mode`, `foundation mode` の分岐を作る。
- 点数評価より、審査観点ごとの「読者が信じる根拠」の確認に寄せる。

### `budget-planning`

良い点:

- 年度配分、積算根拠、一式回避などは有用。

不足:

- 申請予算と採択後執行の論点が混ざる。
- 科研費の費目・設備備品・間接経費の説明は、年度・制度・機関処理で変わるため固定記述に注意が必要。

改修:

- `申請時の予算計画` に限定し、採択後は `budget-execution` へ渡す。
- 出力に「公募要領で確認した予算ルール」「所属機関に確認する会計論点」を分ける。

### `budget-execution`

良い点:

- 研究目的、資金ルール、機関ルール、証憑、期限を分けており、実務に近い。

不足:

- 補助金、委託費、共同研究費、寄附金、民間助成で処理が変わる。
- 検収、発注権限、利益相反、競争的研究費の統一ルールとの接続が不足。

改修:

- 資金種別ごとの「確認すべき規程」を追加。
- 発注前確認メモを標準出力にする。

### `progress-report`

良い点:

- 成果一覧、進捗、計画との差異分析は報告書作成に合う。

不足:

- ポストアワード実務では、報告書作成は日常の進捗管理・成果管理・定例会議の結果として出る。現行は報告時点の文書作成に偏る。

改修:

- `project-progress-tracker` と連携し、月次進捗ログから報告書を生成する流れを追加。
- 評価委員会、ヒアリング、外部有識者会議への対応を追加。

### `research-metrics`

良い点:

- OpenAlex、researchmap、KAKEN など公開情報源を含めている。
- 指標の限界を明示している。

不足:

- 個人の業績整理と機関 IR は成果物・リスクが異なる。
- Scopus / WoS / SciVal など契約 DB は権限がある場合に限る。

改修:

- 個人支援版と組織 IR 版の出力を分離。
- 同姓同名、所属異動、分野補正、自己引用の扱いをチェックリスト化。

### `era-guide`

良い点:

- e-Rad のよくある操作とトラブルをまとめている。

不足:

- e-Rad は UI・時間・ファイル上限が変わるため、固定値が危険。
- 実際の操作は権限ロール、所属機関設定、公募に依存する。

改修:

- 公式マニュアル・公募要領・所属機関通知の確認を第一手順にする。
- 操作手順は「確認すべき画面・項目」の形式に寄せる。

### `ethics-review-support`

良い点:

- 審査要否を断定せず、要確認を残す姿勢はよい。
- 対象者保護、同意、データ管理が入っている。

不足:

- 医学系、生命科学系、社会科学系、教育実践、二次データ利用で適用ルールが異なる。
- 倫理審査、個人情報、契約、情報セキュリティが混ざりやすい。

改修:

- 「要否判定」より「倫理審査窓口へ確認する論点整理」を中心にする。
- 指針対象外でも個人情報・契約・情報セキュリティ確認が残ることを明示。

### `research-data-management`

良い点:

- DMP を運用タスクへ落とす方針がよい。
- 公開/制限付き公開/非公開/要確認の判定が実務的。

不足:

- 公的資金の研究データ管理・利活用方針、メタデータ、機関リポジトリ、研究データ基盤への参照が必要。

改修:

- DMP の確認ソースに内閣府研究 DX、所属機関ポリシー、FA 公募要領を追加。
- 公開しない理由を説明可能にするテンプレートを追加。

### `event-organizer`

良い点:

- 企画書、告知文、アンケート、当日進行など成果物が具体的。

不足:

- Luma + Zoom 無料開催に固定され、URA スキル標準のイベント開催関連全体をカバーしない。
- 学会、成果報告会、外部評価委員会、産学連携イベント、展示会の運営とは違う。

改修:

- `event-organizer` は汎用イベント開催関連へ広げる。
- 現行内容は `online-study-session-organizer` として分離してもよい。

### `create-ra-skill`

良い点:

- 暗黙知を SKILL.md にする入口として有用。

不足:

- 生成後の実務監査、公式ソース確認、URA 22 業務への対応付けがない。

改修:

- ヒアリングに「対応する URA 業務区分」「公式ソース」「機関差」「要確認事項」を追加。
- 生成後にこの監査基準で自己レビューさせる。
